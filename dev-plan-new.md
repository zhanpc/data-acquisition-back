# 数据采集系统开发落地计划（dev-plan-new）

## 1. 文档说明

- 本文档以当前仓库中的代码、配置文件、SQL 初始化脚本为事实依据，用于梳理项目当前实现状态与后续落地路径。
- [design-enhanced.md](/E:/workspace-idea/data-acquisition-back/design-enhanced.md) 代表目标设计方向，不等于当前版本全部已实现能力。
- 本文档用于后续开发落地、阶段跟踪和验收，不用于描述具体代码修改细节。
- 本文档重点说明三类内容：当前已实现功能、已有基础但尚未完成的功能、以及尚未开始实现的目标能力。
- 本文档不改动任何现有代码、SQL、配置和设计文件，仅作为当前项目的开发落地基线文档。

## 2. 当前项目实现概况

当前项目已经具备工业数据采集系统的基础框架，后端基于 Spring Boot 构建，已接入 MySQL 作为配置数据库、TDengine 作为时序数据库、Kafka 作为消息通道，并引入了 Redis 相关基础组件。协议侧已具备 IEC 104、Modbus TCP、Modbus RTU 三类基础接入能力，数据链路已形成“配置加载 -> 建立连接 -> 协议采集 -> WAL 本地落盘 -> Kafka -> TDengine”的基本闭环。当前系统管理接口已具备登录、用户、角色、菜单管理能力，但业务采集管理接口尚未完整落地。整体来看，当前版本已形成“基础采集框架可运行、业务交付尚未闭环”的状态，距离目标架构在配置模型、查询封装、可靠性治理、运维能力和高级架构能力方面仍有明显差距。

| 项目项 | 当前现状 |
|------|------|
| 技术栈 | Spring Boot + MySQL + TDengine + Kafka + Redis（已引入基础组件） |
| 当前协议 | IEC 104、Modbus TCP、Modbus RTU |
| 当前数据链路 | 配置加载 -> 建立连接 -> 协议采集 -> WAL -> Kafka -> TDengine |
| 当前接口现状 | 系统管理接口已具备，业务采集管理接口未完整落地 |
| 当前项目定位 | 基础采集框架已形成，但尚未达到可完整交付的业务平台状态 |

## 3. 已实现功能

### 3.1 基础工程能力

- 已建立基于 Spring Boot 的后端工程基础框架，具备标准化的启动入口、依赖管理和组件扫描能力。
- 已配置多环境配置文件，包含 `application.yml`、`application-test.yml`、`application-dev.yml`、`application-local.yml`、`application-prod.yml`。
- 已接入 Swagger / Springdoc 文档能力，可对现有 REST 接口进行基础说明。
- 已提供 `Dockerfile`、`Dockerfile.test`、`Dockerfile.prod`、`docker-compose.yml` 等部署相关文件，具备基础容器化部署条件。

当前边界说明：当前工程基础已具备，但部署说明、环境初始化步骤和面向业务的运行规范尚未形成完整交付文档。

### 3.2 配置管理基础能力

- 已建立 MySQL 配置库初始化脚本，包含 `station_config`、`point_config`、`alarm_rule`、`alarm_history` 等核心基础表。
- 已实现 `ConfigService`，支持应用启动后从 MySQL 加载场站配置和测点配置。
- 已实现 `ConfigCache`，支持场站配置、测点配置的内存缓存，减少运行期频繁查库。
- 已实现 `ConnectionInitializer`，在配置加载完成后自动建立场站连接。
- `StationConfig` 已具备 `extraParams` 字段，可为 Modbus RTU 等协议承载扩展参数。
- 协议配置枚举已统一为 `IEC_104`、`MODBUS_TCP`、`MODBUS_RTU`，初始化 SQL 与当前代码口径已对齐。

当前边界说明：当前缓存实现以应用内存为主，并非设计目标中的 Redis 事件驱动热加载体系；配置模型仍偏基础版，尚未覆盖采集频率、点位类型、缩放系数、阈值、补传策略等最终设计所需字段。

### 3.3 协议采集能力

#### IEC 104

- 已实现 IEC 104 协议处理服务，支持按场站参数建立连接。
- 已支持接收 ASDU 数据并解析信息对象。
- 已支持根据 IOA 在缓存中查找测点配置并转换为统一 `DataPoint`。
- 已支持总召唤能力。
- 已支持单点命令下发。
- 已支持设点命令下发。

#### Modbus TCP

- 已实现 Modbus TCP 协议处理服务，支持建立 TCP 连接。
- 已支持保持寄存器读取。
- 已支持输入寄存器读取。
- 已支持线圈读取与离散输入读取。
- 已支持单寄存器写入、单线圈写入、多寄存器写入。
- 已支持将读取结果转换为统一 `DataPoint` 并推送到数据链路。

#### Modbus RTU

- 已实现 Modbus RTU 协议处理服务，支持基于串口参数建立连接。
- 已支持保持寄存器读取。
- 已支持输入寄存器读取。
- 已支持线圈读取与离散输入读取。
- 已支持单寄存器写入。
- 已支持单线圈写入、多寄存器写入。
- 已支持将读取结果转换为统一 `DataPoint` 并推送到数据链路。

当前边界说明：当前协议层具备基础接入能力，Modbus 已具备按测点配置驱动的基础自动轮询能力，RTU 也已纳入统一轮询与重连闭环；但当前仍缺少按寄存器区间合并、按采集频率分层调度等优化能力，协议主要差距也集中在调度粒度与运行治理，而非枚举口径不统一。

### 3.4 数据链路能力

- 已定义统一数据模型 `DataPoint`，用于承载不同协议的采集结果。
- 已实现 `WalManager`，支持采集数据本地落盘和应用启动后的恢复。
- 已实现 `DataProducer`，支持将采集数据写入 WAL 后发送到 Kafka。
- 已实现 `DataConsumer`，支持从 Kafka 批量消费数据并写入 TDengine。
- 已实现 `TDengineService`，支持基于测点配置中的表名进行批量写入。
- 已实现最新值、历史值、聚合查询的基础服务方法。
- 已建立 TDengine 初始化脚本，包含 IEC 104 和 Modbus 的超级表及测试子表。

当前边界说明：当前 Kafka 只有单 Topic 链路，当前 WAL 不是 30 天缓存体系；WAL 恢复也仍停留在基础版，尚未形成断点记录、恢复幂等、恢复后旧文件有序清理的完整闭环；当前 TDengine 已存在最新值、历史值、聚合查询的基础方法，但查询逻辑仍偏向 IEC 104 超级表，尚未覆盖多协议统一查询模型。

### 3.5 系统管理能力

- 已实现登录接口。
- 已实现用户管理接口。
- 已实现角色管理接口。
- 已实现菜单管理接口。
- 已具备 JWT、拦截器、异常处理等系统管理基础能力。

当前边界说明：当前系统管理模块已形成基础后台能力，但它不等于业务采集管理模块，场站配置、测点配置、采集状态、告警查询等业务接口仍未完整落地。

### 3.6 测试与初始化脚本基础能力

- 已提供 MySQL 初始化脚本和 TDengine 初始化脚本，可用于基础环境准备。
- 已存在 IEC 104、Modbus TCP 的 `SpringBootTest` 集成测试样例，用于联调阶段验证基础协议动作。
- 已通过项目编译，说明当前主干代码在依赖层面可构建。

当前边界说明：当前测试更偏联调样例，依赖真实设备或特定环境参数，尚未形成稳定、可重复执行的自动化测试基线；交付所需的测试用例、联调说明、部署说明仍需补充。

## 4. 待实现功能

本章定义的“待实现功能”是指：仓库中已经存在部分基础代码或数据模型，但尚不能满足最终交付要求，需要继续完善后才能形成可交付能力。

### 4.1 采集任务调度完善

- 当前系统已具备协议连接、单次读写和基础自动采集调度能力，但距离完整可交付的持续采集调度主流程仍有差距。
- Modbus TCP 当前已具备按测点配置驱动的统一自动轮询机制，但仍缺少寄存器区间优化、频率分层和更细粒度的任务编排能力。
- Modbus RTU 当前已接入统一轮询、统一发送与基础状态管理，但串口场景下的异常治理、调度优化和更强运行保障仍需完善。
- IEC 104 已实现建连后自动总召和异常重连，但缺少更完整的策略配置、状态可视化和治理规范。

当前现状说明：协议能力不是没有，而是缺少从“能连、能读”升级到“可持续自动采集”的调度闭环。

### 4.2 时序查询能力完善

- 当前 TDengine 写入逻辑按 `tableName` 动态写入，说明多协议分表模型已经具备基础。
- 当前已存在最新值、历史值、聚合查询的基础方法，但查询逻辑仍偏向 IEC 104 超级表，尚未形成多协议统一查询能力。
- 后续需要将读写模型统一为面向业务查询的服务能力，而不是仅支持部分测试查询路径。

当前现状说明：当前数据不是不能入库，而是查询能力尚未完成统一封装，无法支撑多协议、多测点的完整业务查询。

### 4.3 业务配置接口完善

- 当前系统管理接口已存在，但业务配置接口尚未成型。
- 缺少场站配置管理接口。
- 缺少测点配置管理接口。
- 缺少告警规则管理接口。
- 缺少采集连接状态、运行状态、最近采集时间等运维类接口。

当前现状说明：配置数据表和缓存加载逻辑已经存在，但缺少面向业务使用和运维管理的完整 API 层。

### 4.4 可靠性机制补强

- 当前已具备 WAL、本地恢复、WAL 启动恢复、Kafka 手动提交等基础可靠性手段。
- 采集发送失败后的重试闭环仍不完整，失败后更多停留在日志级处理。
- WAL 当前缺少更完善的文件滚动、恢复顺序控制、幂等去重和生命周期管理。
- WAL 启动恢复当前仍需补齐“恢复后旧文件清理”和“避免重复补发”的安全闭环。
- Kafka 消费异常后的补偿、回退、重试和状态记录机制仍需补齐。

当前现状说明：系统不是没有可靠性设计，而是当前可靠性能力仍处于基础版；目前已具备 Kafka 手动提交、WAL 启动恢复与失败后保留 WAL 文件的基础能力，但补偿、幂等、回退和生命周期治理仍未补齐，尚不能支撑最终交付级运行要求。

### 4.5 告警能力补齐

- 当前已具备告警规则表和告警历史表基础结构。
- 当前尚未形成完整的阈值判断、告警触发、告警写库和告警查询闭环。
- 告警能力尚未与采集链路、查询链路和运维展示结合。

当前现状说明：告警并非完全空白，而是目前只有基础表结构，规则执行和业务闭环尚未形成。

## 5. 未实现功能

本章定义的“未实现功能”是指：目标设计中已提出，但当前仓库中尚无有效实现入口或尚无可交付基础代码的能力。

### 5.1 OPC UA / OPC DA 接入

- 目标设计中已将 OPC UA / OPC DA 作为重要协议能力之一。
- 当前代码仓库中未发现 OPC 协议处理服务、配置模型、连接初始化逻辑和数据采集链路实现。

当前未实现原因/现状：目前项目仅实现 IEC 104、Modbus TCP、Modbus RTU 三类协议，OPC 仍停留在目标设计阶段。

### 5.2 Kafka 双通道架构

- 目标设计要求区分实时通道和历史补传通道。
- 当前实现只有单 Topic `iot-data`，未区分实时数据和历史补传数据。
- 也没有围绕双 Topic 构建独立消费策略、优先级控制和补传限流机制。

当前未实现原因/现状：当前消息链路仍是基础单通道架构，尚未进入双通道设计的实现阶段。

### 5.3 断点续传断点表与后台补传

- 目标设计中要求存在断点记录表、补传状态、后台补传任务和补传进度管理。
- 当前项目虽然有 WAL 恢复能力，但没有断点记录表，也没有独立的历史补传任务框架。

当前未实现原因/现状：当前可靠性手段仍以“本地落盘 + 启动恢复”为主，尚未形成真正的断点续传体系。

### 5.4 多通道数据融合与故障切换

- 目标设计中定义了多通道数据融合、优先级切换和故障切换。
- 当前仓库中尚无融合引擎、优先级策略实现、通道状态仲裁或切换逻辑。

当前未实现原因/现状：当前系统仍以单场站单协议采集链路为主，尚未进入多源融合场景。

### 5.5 边缘数据处理引擎

- 目标设计中提出了数据质量校验、阈值监测、异常检测和修正建议。
- 当前代码中尚未形成独立的边缘处理引擎，也没有质量评估、异常识别、修正建议等处理链路。

当前未实现原因/现状：当前系统重点仍在打通基础采集链路，边缘智能处理能力尚未开始落地。

### 5.6 配置热加载

- 目标设计中要求支持运行期配置热加载。
- 当前配置加载主要发生在应用启动阶段，尚未形成运行期变更后自动刷新连接、缓存和采集任务的机制。

当前未实现原因/现状：当前配置体系以启动加载为主，热加载机制仍未实现。

### 5.7 遥控审计与命令闭环

- 目标设计中要求具备遥控命令下发、执行反馈、审计留痕和闭环管理。
- 当前代码已存在部分遥控命令发送能力，但缺少命令队列、执行反馈记录、审计日志和统一闭环状态管理。

当前未实现原因/现状：遥控基础动作已具备，但审计和闭环体系尚未建立，因此仍不能视为完整实现。

### 5.8 完整可观测性与运维监控

- 目标设计中要求具备采集状态监控、消息堆积监控、WAL 容量监控、补传进度监控和关键运行指标展示。
- 当前项目虽引入了 Actuator 等基础组件，但尚未形成完整面向采集业务的可观测性体系。

当前未实现原因/现状：当前监控更多停留在基础框架层，尚未形成面向协议采集和数据链路的专项运维能力。

## 6. 分阶段实施计划

### Phase 1：建立当前版本可交付基线

**目标**

- 先让现有链路自洽，统一配置口径，补齐基础业务查询、配置接口和交付文档，使当前版本具备基础交付条件。

**主要任务**

- 补齐并校准配置模型，明确场站、测点、协议扩展参数与运行状态字段的边界。
- 补齐采集调度主流程，重点完善现有 Modbus TCP、Modbus RTU 的自动轮询机制，以及 IEC 104 的初始化采集策略。
- 修正 TDengine 查询模型，统一最新值、历史值、聚合查询能力。
- 增加基础业务 API，包括场站配置、测点配置、告警规则、采集状态查询等接口。
- 补充基础交付文档，包括测试用例、联调说明、部署初始化说明和基础验收说明。
- 建立单协议到多协议的基础联调和验收链路。

**输出结果**

- 配置口径统一，可维护。
- 协议链路可运行、可自动采集。
- 采集数据可查询。
- 基础联调与交付文档可用。
- 当前版本具备基础交付能力。

### Phase 2：补齐可靠性与运行保障

**目标**

- 让系统从“能跑”提升到“能稳定运行”，补齐异常恢复、基础补偿和运行可观测能力。

**主要任务**

- 增强 WAL 机制，完善文件滚动、恢复顺序、保留策略和幂等处理。
- 修复 WAL 恢复后的旧文件清理与重复补发风险，形成可验证的恢复闭环。
- 补齐 Kafka 和 TDengine 异常处理闭环，完善失败重试与补偿逻辑。
- 实现断点续传基础版，包括断点记录、补传任务、补传状态管理。
- 增加基础告警处理能力，使阈值规则能够触发并落历史记录。
- 增加运行状态和监控接口，支持查看连接状态、最近采集状态、关键链路运行情况。
- 补充异常场景测试、恢复验证记录和运维排障文档。

**输出结果**

- 异常可恢复。
- 运行状态可观测。
- 告警可追踪。
- 数据可靠性明显提升。

### Phase 3：推进目标架构高级能力

**目标**

- 在前两阶段稳定的基础上，逐步向 [design-enhanced.md](/E:/workspace-idea/data-acquisition-back/design-enhanced.md) 的目标架构靠拢。

**主要任务**

- 接入 OPC UA / OPC DA。
- 引入 Kafka 双通道架构，拆分实时链路与历史补传链路。
- 实现历史补传机制和断点续传增强版。
- 实现多通道数据融合与故障切换。
- 建立边缘数据处理引擎。
- 增加遥控审计、命令闭环和配置热加载能力。

**输出结果**

- 架构能力接近目标设计。
- 系统具备更强扩展性。
- 系统具备高可靠运行基础。

## 7. 各阶段验收标准

### Phase 1 验收标准

- 场站配置、测点配置可正确加载并缓存。
- 协议连接可正确建立。
- 采集数据可进入 Kafka。
- Kafka 数据可写入 TDengine。
- 最新值、历史值查询可用。
- 基础业务配置接口完成落地后，可用于维护场站和测点配置。
- 至少具备一套可重复执行的基础联调步骤和测试用例说明。

### Phase 2 验收标准

- Kafka 故障或 TDengine 故障后链路具备恢复能力。
- WAL 恢复逻辑可通过测试验证。
- WAL 恢复后不会因旧文件残留导致重复补发失控。
- 告警规则可以生效并记录历史。
- 运行状态、采集状态可通过接口或监控视图查看。
- 基础补传逻辑可执行并可跟踪状态。

### Phase 3 验收标准

- 新协议接入后可通过统一配置接入系统。
- 历史补传不会影响实时链路。
- 多通道融合能力可执行切换。
- 关键运维指标可查看。
- 高级能力与目标设计文档保持一致。

## 8. 风险与依赖

- 调度与运行状态虽已具备基础闭环，但缺少细粒度调度、统一查询与更强可靠性治理，仍会影响 Phase 1 的交付质量。
- 当前 Modbus 轮询仍以“逐点、单地址读取”为主，配置量上升后会直接影响吞吐与扩展性。
- 当前 `ConfigCache` 查询大量依赖遍历匹配，随着场站和测点规模上升，可能成为运行期热点。
- TDengine 表模型与查询模型不一致，会直接影响业务查询能力落地。
- WAL 当前虽有基础恢复能力，但恢复后旧文件清理与幂等控制不完整，存在重复补发和恢复结果不确定风险。
- Kafka 可靠性策略当前与目标设计有差距，若不补齐，系统难以支撑更高可靠性要求。
- OPC、双通道、边缘处理等模块体量较大，需独立排期，不宜与基础稳定性改造同时并行推进。
- 现有系统管理模块不等于业务采集管理模块，若不补齐业务接口，配置和运维能力无法真正落地。
- 断点续传、补传、融合切换等能力对配置模型、状态管理和运维监控存在较强依赖，需要在前置阶段先补齐基础能力。
- 当前测试主要依赖真实设备或环境参数，自动化验证不足，会增加后续改造回归成本和交付风险。

## 9. 实施顺序建议

1. 先统一配置模型、业务口径和初始化文档。
2. 再补采集调度与查询能力，先解决“能稳定采、能稳定查”。
3. 再补业务 API、运行状态接口和基础告警闭环。
4. 再做 WAL、Kafka、补传相关可靠性增强与异常场景验证。
5. 最后做 OPC、双通道、融合、边缘处理和热加载等高级能力。
